# Техническое задание (ТЗ) — CarHub (Автомобильный портал)

### 1. Цель проекта
Создать веб-портал для размещения и поиска объявлений о продаже автомобилей с ролевой моделью доступа, валидацией бизнес-логики, REST API и оптимизированными запросами.

### 2. Роли пользователей и их возможности (минимум 3 пункта на роль)

#### 2.1. Обычный пользователь (user / buyer)
1. Просматривать активные объявления (список + детальная страница).
2. Добавлять/удалять объявления в личное "Избранное".
3. Регистрироваться и входить в систему (с указанием телефона).

#### 2.2. Продавец (seller)
1. Создавать новые объявления (автоматически уходят на модерацию, статус `moderation`).
2. Редактировать и удалять **только свои** объявления.
3. Переводить своё объявление в статус `sold` (пометить как проданное).

#### 2.3. Модератор (moderator)
1. Просматривать объявления, находящиеся на модерации.
2. Одобрять (переводить `moderation` → `active`) или отклонять объявления.
3. Просматривать все объявления независимо от статуса (через админку или API).

#### 2.4. Администратор (admin)
1. Полный доступ ко всем объявлениям, пользователям и моделям (CRUD).
2. Использовать импорт/экспорт объявлений (в т.ч. кастомный XLSX).
3. Управлять пользователями, брендами, моделями и просматривать историю изменений (simple-history).

### 3. Бизнес-логика и валидация (обязательная к реализации)

- При создании объявления обычным пользователем/продавцом статус **всегда** устанавливается в `moderation` (нельзя сразу опубликовать).
- Обычный пользователь/продавец может перевести своё объявление **только** в `sold`. Переход обратно в `active` или `moderation` запрещён (только через модератора/админа).
- Модель автомобиля должна принадлежать выбранной марке (валидация на уровне формы и сериализатора).
- Цена не может быть ≤ 0.
- Год выпуска: 1900 ≤ year ≤ текущий год + 1.
- При удалении старых проданных объявлений (> 1 года) используется management-команда.
- Избранное доступно только для активных объявлений (для обычных пользователей).

### 4. Оптимизация запросов
- Обязательное использование `select_related('brand', 'model', 'user')` при получении списков и деталей автомобилей.
- `prefetch_related('photos')` при необходимости загрузки доп. фотографий.

### 5. REST API (для Postman + сериализаторов)
- Эндпоинты: `/api/cars/`, `/api/brands/`
- Поддержка фильтрации через django-filter FilterSet (цена, бренд, год и т.д.).
- Кастомные действия: increment views, cheap cars.
- Сериализаторы должны демонстрировать:
  - `SerializerMethodField`
  - Передачу данных через `context` (например, `is_favorited` для текущего пользователя)

### 6. Дополнительные вычисления (аннотации)
- Количество фотографий у объявления (`photo_count`).
- Количество активных объявлений у продавца.
- Примеры агрегаций по брендам (средняя цена и т.д.).

### 7. Фильтрация
- Реализован `CarFilter` (django_filters.FilterSet) с:
  - `brand`, `model`
  - `price` (диапазон)
  - `year` (диапазон или min/max)
  - `status`
